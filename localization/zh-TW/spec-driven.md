# 規格驅動開發 (SDD)

## 權力反轉

數十年來，程式碼一直是核心。規格服務於程式碼——它們是我們在「真正的工作」開始後建立並丟棄的鷹架。我們編寫 PRD 來指導開發，建立設計文件來告知實作，繪製圖表來視覺化架構。但這些始終從屬於程式碼本身。程式碼是真理。其他一切，充其量只是良好的意圖。程式碼是真理的來源，隨著它的推進，規格很少能跟上。由於資產（程式碼）和實作是一體的，因此在不嘗試從程式碼建構的情況下，很難進行平行實作。

規格驅動開發 (SDD) 顛覆了這種權力結構。規格不服務於程式碼——程式碼服務於規格。(產品需求文件-規格) PRD 不是實作的指南；它是產生實作的來源。技術計畫不是告知編碼的文件；它們是產生程式碼的精確定義。這不是我們建構軟體方式的增量改進。這是對驅動開發的根本性重新思考。

規格與實作之間的差距自軟體開發誕生以來一直困擾著它。我們試圖透過更好的文件、更詳細的需求、更嚴格的流程來彌補它。這些方法之所以失敗，是因為它們接受了這種差距是不可避免的。它們試圖縮小它，但從未消除它。SDD 透過使規格及其從規格中誕生的具體實作計畫可執行來消除這種差距。當規格到實作計畫產生程式碼時，沒有差距——只有轉換。

這種轉換現在之所以可能，是因為 AI 可以理解和實作複雜的規格，並建立詳細的實作計畫。但是，沒有結構的原始 AI 產生會導致混亂。SDD 透過規格和隨後的實作計畫提供這種結構，這些計畫足夠精確、完整和明確，足以產生可運作的系統。規格成為主要構件。程式碼成為其在特定語言和框架中的表達（作為實作計畫中的實作）。

在這個新世界中，維護軟體意味著演進規格。開發團隊的意圖以自然語言（「**意圖驅動開發**」）、設計資產、核心原則和其他準則表達。開發的**通用語**轉向更高層次，程式碼是最後一哩路徑。

偵錯意味著修復產生不正確程式碼的規格及其實作計畫。重構意味著為了清晰而重新組織。整個開發工作流程圍繞規格作為中央真理來源重新組織，實作計畫和程式碼作為持續重新產生的輸出。使用新功能更新應用程式或建立新的平行實作，因為我們是富有創造力的人，這意味著重新審視規格並建立新的實作計畫。因此，這個過程是 0 -> 1, (1', ..), 2, 3, N。

開發團隊專注於他們的創造力、實驗和批判性思維。

## SDD 工作流程實踐

工作流程始於一個想法——通常是模糊且不完整的。透過與 AI 的迭代對話，這個想法成為一個全面的 PRD。AI 會提出澄清問題，識別邊緣案例，並幫助定義精確的驗收標準。在傳統開發中可能需要數天的會議和文件，在數小時的專注規格工作中即可完成。這改變了傳統的 SDLC——需求和設計成為持續的活動，而不是離散的階段。這支援**團隊流程**，即團隊審查的規格以分支表達和版本化，並合併。

當產品經理更新驗收標準時，實作計畫會自動標記受影響的技術決策。當架構師發現更好的模式時，PRD 會更新以反映新的可能性。

在整個規格過程中，研究代理程式會收集關鍵上下文。他們調查函式庫相容性、效能基準和安全隱患。組織限制會自動發現並應用——您公司的資料庫標準、驗證要求和部署策略無縫整合到每個規格中。

從 PRD 中，AI 會產生將需求映射到技術決策的實作計畫。每個技術選擇都有文件化的理由。每個架構決策都可以追溯到特定需求。在整個過程中，一致性驗證持續提高品質。AI 會分析規格中的模糊性、矛盾和差距——這不是一次性的關卡，而是持續的精煉。

程式碼產生在規格及其實作計畫足夠穩定時立即開始，但它們不必「完整」。早期的產生可能是探索性的——測試規格在實踐中是否有意義。領域概念成為資料模型。使用者故事成為 API 端點。驗收情境成為測試。這透過規格將開發和測試合併——測試情境不是在程式碼之後編寫的，它們是產生實作和測試的規格的一部分。

回饋循環超越了初始開發。生產指標和事件不僅觸發熱修復——它們還更新規格以進行下一次重新產生。效能瓶頸成為新的非功能性需求。安全漏洞成為影響所有未來世代的限制。規格、實作和操作現實之間的這種迭代舞蹈是真正理解出現的地方，也是傳統 SDLC 轉變為持續演進的地方。

## 為什麼 SDD 現在很重要

三個趨勢使 SDD 不僅可能而且必要：

首先，AI 能力已達到一個閾值，自然語言規格可以可靠地產生可運作的程式碼。這不是要取代開發人員——而是透過自動化從規格到實作的機械翻譯來放大他們的效率。它可以放大探索和創造力，它可以輕鬆支援「重新開始」，它支援加減法和批判性思維。

其次，軟體複雜性持續呈指數級增長。現代系統整合了數十種服務、框架和依賴項。透過手動流程使所有這些部分與原始意圖保持一致變得越來越困難。SDD 透過規格驅動產生提供系統性對齊。框架可能會演進以提供 AI 優先支援，而不是人類優先支援，或圍繞可重複使用的元件進行架構。

第三，變化速度加快。如今，需求變化比以往任何時候都快得多。轉向不再是例外——這是預期的。現代產品開發要求根據使用者回饋、市場條件和競爭壓力快速迭代。傳統開發將這些變化視為中斷。每次轉向都需要手動透過文件、設計和程式碼傳播變化。結果要麼是緩慢、謹慎的更新，限制了速度，要麼是快速、魯莽的變化，累積了技術債務。

SDD 可以支援假設/模擬實驗，「如果我們需要重新實作或更改應用程式以促進銷售更多 T 恤的業務需求，我們將如何實作和實驗？」

SDD 將需求變化從障礙轉變為正常工作流程。當規格驅動實作時，轉向成為系統性重新產生，而不是手動重寫。更改 PRD 中的核心需求，受影響的實作計畫會自動更新。修改使用者故事，相應的 API 端點會重新產生。這不僅僅是關於初始開發——這是關於透過不可避免的變化來維持工程速度。

## 核心原則

**規格作為通用語**：規格成為主要構件。程式碼成為其在特定語言和框架中的表達。維護軟體意味著演進規格。

**可執行規格**：規格必須足夠精確、完整和明確，才能產生可運作的系統。這消除了意圖與實作之間的差距。

**持續精煉**：一致性驗證持續進行，而不是一次性關卡。AI 會持續分析規格中的模糊性、矛盾和差距。

**研究驅動上下文**：研究代理程式在整個規格過程中收集關鍵上下文，調查技術選項、效能影響和組織限制。

**雙向回饋**：生產現實告知規格演進。指標、事件和操作學習成為規格精煉的輸入。

**探索性分支**：從相同的規格產生多種實作方法，以探索不同的最佳化目標——效能、可維護性、使用者體驗、成本。

## 實作方法

如今，實踐 SDD 需要組裝現有工具並在整個過程中保持紀律。該方法可以透過以下方式實踐：

- 用於迭代規格開發的 AI 助理
- 用於收集技術上下文的研究代理程式
- 用於將規格轉換為實作的程式碼產生工具
- 適用於規格優先工作流程的版本控制系統
- 透過 AI 分析規格文件進行一致性檢查

關鍵在於將規格視為真理的來源，程式碼是服務於規格的產生輸出，而不是反過來。

## 透過命令簡化 SDD

SDD 方法透過三個強大的命令顯著增強，這些命令自動化了規格 → 規劃 → 任務工作流程：

### `/specify` 命令

此命令將簡單的功能描述（使用者提示）轉換為完整、結構化的規格，並自動進行儲存庫管理：

1. **自動功能編號**：掃描現有規格以確定下一個功能編號（例如，001、002、003）
2. **分支建立**：從您的描述產生語義分支名稱並自動建立
3. **基於範本的產生**：複製並使用您的需求自訂功能規格範本
4. **目錄結構**：為所有相關文件建立適當的 `specs/[branch-name]/` 結構

### `/plan` 命令

一旦功能規格存在，此命令會建立一個全面的實作計畫：

1. **規格分析**：讀取並理解功能需求、使用者故事和驗收標準
2. **核心準則合規性**：確保與專案核心準則和架構原則保持一致
3. **技術翻譯**：將業務需求轉換為技術架構和實作細節
4. **詳細文件**：產生資料模型、API 契約和測試情境的支援文件
5. **快速入門驗證**：產生一個快速入門指南，捕捉關鍵驗證情境

### `/tasks` 命令

建立計畫後，此命令會分析計畫和相關設計文件，以產生可執行的任務清單：

1.  **輸入**：讀取 `plan.md`（必需），如果存在，還讀取 `data-model.md`、`contracts/` 和 `research.md`
2.  **任務衍生**：將契約、實體和情境轉換為特定任務
3.  **平行化**：標記獨立任務 `[P]` 並概述安全的平行組
4.  **輸出**：在功能目錄中寫入 `tasks.md`，準備好由任務代理程式執行

### 範例：建構聊天功能

以下是這些命令如何改變傳統開發工作流程：

**傳統方法：**

```text
1. 編寫文件中的 PRD (2-3 小時)
2. 建立設計文件 (2-3 小時)
3. 手動設定專案結構 (30 分鐘)
4. 編寫技術規格 (3-4 小時)
5. 建立測試計畫 (2 小時)
總計：約 12 小時的文件工作
```

**帶有命令的 SDD 方法：**

```bash
# 步驟 1：建立功能規格 (5 分鐘)
/specify 具有訊息歷史記錄和使用者狀態的即時聊天系統

# 這會自動：
# - 建立分支「003-chat-system」
# - 產生 specs/003-chat-system/spec.md
# - 使用結構化需求填充它

# 步驟 2：產生實作計畫 (5 分鐘)
/plan 用於即時訊息的 WebSocket，用於歷史記錄的 PostgreSQL，用於狀態的 Redis

# 步驟 3：產生可執行任務 (5 分鐘)
/tasks

# 這會自動建立：
# - specs/003-chat-system/plan.md
# - specs/003-chat-system/research.md (WebSocket 函式庫比較)
# - specs/003-chat-system/data-model.md (訊息和使用者綱要)
# - specs/003-chat-system/contracts/ (WebSocket 事件，REST 端點)
# - specs/003-chat-system/quickstart.md (關鍵驗證情境)
# - specs/003-chat-system/tasks.md (從計畫衍生的任務清單)
```

在 15 分鐘內，您將擁有：

- 包含使用者故事和驗收標準的完整功能規格
- 包含技術選擇和理由的詳細實作計畫
- 準備好用於程式碼產生的 API 契約和資料模型
- 用於自動化和手動測試的全面測試情境
- 所有文件都已在功能分支中正確版本化

### 結構化自動化的力量

這些命令不僅節省時間，還強制執行一致性和完整性：

1. **沒有遺漏的細節**：範本確保考慮到每個方面，從非功能性需求到錯誤處理
2. **可追溯的決策**：每個技術選擇都追溯到特定需求
3. **活文件**：規格與程式碼保持同步，因為它們產生程式碼
4. **快速迭代**：在幾分鐘而不是幾天內更改需求並重新產生計畫

這些命令透過將規格視為可執行構件而不是靜態文件來體現 SDD 原則。它們將規格過程從必要的邪惡轉變為開發的驅動力。

### 範本驅動的品質：結構如何約束 LLM 以獲得更好的結果

這些命令的真正力量不僅在於自動化，還在於範本如何引導 LLM 行為以獲得更高品質的規格。範本充當複雜的提示，以有效的方式約束 LLM 的輸出：

#### 1. **防止過早的實作細節**

功能規格範本明確指示：

```text
- ✅ 專注於使用者需要什麼以及為什麼
- ❌ 避免如何實作（沒有技術堆疊、API、程式碼結構）
```

此約束強制 LLM 維護適當的抽象層次。當 LLM 可能自然地跳到「使用 React 和 Redux 實作」時，範本會使其專注於「使用者需要即時更新其資料」。這種分離確保即使實作技術發生變化，規格也能保持穩定。

#### 2. **強制明確的不確定性標記**

兩個範本都要求使用 `[NEEDS CLARIFICATION]` 標記：

```text
從使用者提示建立此規格時：
1. **標記所有模糊之處**：使用 [NEEDS CLARIFICATION: specific question]
2. **不要猜測**：如果提示沒有指定某些內容，請標記它
```

這可以防止 LLM 做出看似合理但可能不正確的假設的常見行為。LLM 不會猜測「登入系統」使用電子郵件/密碼驗證，而是必須將其標記為 `[NEEDS CLARIFICATION: auth method not specified - email/password, SSO, OAuth?]`。

#### 3. **透過清單進行結構化思考**

範本包含全面的清單，作為規格的「單元測試」：

```markdown
### 需求完整性
- [ ] 沒有 [NEEDS CLARIFICATION] 標記
- [ ] 需求可測試且明確
- [ ] 成功標準可衡量
```

這些清單強制 LLM 系統地自我審查其輸出，捕捉否則可能會遺漏的漏洞。這就像給 LLM 一個品質保證框架。

#### 4. **透過關卡進行核心準則合規性**

實作計畫範本透過階段關卡強制執行架構原則：

```markdown
### 階段 -1：實作前關卡
#### 簡潔性關卡 (第七條)
- [ ] 使用 ≤3 個專案？
- [ ] 避免超前設計？
#### 反抽象關卡 (第八條)
- [ ] 直接使用框架？
- [ ] 單一模型表示？
```

這些關卡透過讓 LLM 明確證明任何複雜性來防止過度設計。如果關卡失敗，LLM 必須在「複雜性追蹤」部分記錄原因，為架構決策建立問責制。

#### 5. **分層細節管理**

範本強制執行適當的資訊架構：

```text
**重要**：此實作計畫應保持高層次且可讀。
任何程式碼範例、詳細演算法或廣泛的技術規格
必須放置在適當的 `implementation-details/` 文件中
```

這可以防止規格變成難以閱讀的程式碼轉儲的常見問題。LLM 學會維護適當的細節層次，將複雜性提取到單獨的文件中，同時保持主文件可導航。

#### 6. **測試優先思考**

實作範本強制執行測試優先開發：

```text
### 檔案建立順序
1. 建立 `contracts/` 和 API 規格
2. 依序建立測試檔案：契約 → 整合 → 端對端 → 單元
3. 建立原始檔案以使測試通過
```

此排序約束確保 LLM 在實作之前考慮可測試性和契約，從而產生更穩健和可驗證的規格。

#### 7. **防止推測性功能**

範本明確不鼓勵推測：

```text
- [ ] 沒有推測性或「可能需要」的功能
- [ ] 所有階段都有明確的先決條件和可交付成果
```

這可以阻止 LLM 添加「可有可無」的功能，這些功能會使實作複雜化。每個功能都必須追溯到具有明確驗收標準的具體使用者故事。

### 複合效應

這些約束共同作用，產生以下規格：

- **完整**：清單確保沒有遺漏任何內容
- **明確**：強制澄清標記突出不確定性
- **可測試**：測試優先思考融入流程
- **可維護**：適當的抽象層次和資訊層次結構
- **可實作**：具有具體可交付成果的清晰階段

這些範本將 LLM 從創意寫作者轉變為嚴謹的規格工程師，將其能力引導至產生始終如一的高品質、可執行規格，真正推動開發。

## 核心準則基礎：強制執行架構紀律

SDD 的核心是核心準則——一套管理規格如何成為程式碼的不可變原則。核心準則 (`memory/constitution.md`) 作為系統的架構 DNA，確保每個產生的實作都保持一致性、簡潔性和品質。

### 開發的九條條款

核心準則定義了九條條款，塑造了開發過程的每個方面：

#### 第一條：函式庫優先原則

每個功能都必須以獨立函式庫的形式開始——無一例外。這從一開始就強制執行模組化設計：

```text
Specify 中的每個功能都必須以獨立函式庫的形式存在。
任何功能都不得直接在應用程式程式碼中實作，除非
首先將其抽象為可重複使用的函式庫元件。
```

此原則確保規格產生模組化、可重複使用的程式碼，而不是單體應用程式。當 LLM 產生實作計畫時，它必須將功能結構化為具有清晰邊界和最小依賴項的函式庫。

#### 第二條：CLI 介面授權

每個函式庫都必須透過命令列介面公開其功能：

```text
所有 CLI 介面必須：
- 接受文字作為輸入（透過 stdin、引數或檔案）
- 產生文字作為輸出（透過 stdout）
- 支援 JSON 格式進行結構化資料交換
```

這強制執行可觀察性和可測試性。LLM 不能將功能隱藏在不透明的類別中——一切都必須透過基於文字的介面可存取和可驗證。

#### 第三條：測試優先命令

最具變革性的條款——測試之前沒有程式碼：

```text
這是不可協商的：所有實作都必須遵循嚴格的測試驅動開發。
在以下情況之前，不得編寫任何實作程式碼：
1. 編寫單元測試
2. 測試經使用者驗證和批准
3. 測試確認失敗（紅色階段）
```

這完全顛覆了傳統的 AI 程式碼產生。LLM 不會產生程式碼並希望它能運作，而是必須首先產生定義行為的全面測試，獲得批准，然後才產生實作。

#### 第七條和第八條：簡潔性和反抽象

這兩條條款旨在打擊過度設計：

```text
第 7.3 節：最小專案結構
- 初始實作最多 3 個專案
- 額外專案需要文件證明

第 8.1 節：框架信任
- 直接使用框架功能，而不是包裝它們
```

當 LLM 可能自然地建立精密的抽象時，這些條款強制它證明每一層複雜性。實作計畫範本的「階段 -1 關卡」直接強制執行這些原則。

#### 第九條：整合優先測試

優先於隔離單元測試的實際測試：

```text
測試必須使用真實環境：
- 優先使用真實資料庫而不是模擬
- 使用實際服務實例而不是存根
- 實作前強制執行契約測試
```

這確保產生的程式碼在實踐中運作，而不僅僅是理論上。

### 透過範本強制執行核心準則

實作計畫範本透過具體檢查點來實施這些條款：

```markdown
### 階段 -1：實作前關卡
#### 簡潔性關卡 (第七條)
- [ ] 使用 ≤3 個專案？
- [ ] 避免超前設計？

#### 反抽象關卡 (第八條)
- [ ] 直接使用框架？
- [ ] 單一模型表示？

#### 整合優先關卡 (第九條)
- [ ] 契約已定義？
- [ ] 契約測試已編寫？
```

這些關卡充當架構原則的編譯時檢查。LLM 必須通過關卡或在「複雜性追蹤」部分記錄合理的例外情況，否則無法繼續。

### 不可變原則的力量

核心準則的力量在於其不可變性。雖然實作細節可以演進，但核心原則保持不變。這提供了：

1. **跨時間的一致性**：今天產生的程式碼遵循與明年產生的程式碼相同的原則
2. **跨 LLM 的一致性**：不同的 AI 模型產生架構相容的程式碼
3. **架構完整性**：每個功能都增強而不是破壞系統設計
4. **品質保證**：測試優先、函式庫優先和簡潔性原則確保可維護的程式碼

### 核心準則演進

雖然原則是不可變的，但其應用可以演進：

```text
第 4.2 節：修正案程序
對本核心準則的修改需要：
- 明確記錄變更理由
- 專案維護者審查和批准
- 向後相容性評估
```

這使得該方法能夠在保持穩定性的同時學習和改進。核心準則透過帶有日期的修正案展示了其自身的演進，證明了原則如何根據實際經驗進行精煉。

### 超越規則：一種開發哲學

核心準則不僅僅是一本規則手冊——它是一種塑造 LLM 如何思考程式碼產生的哲學：

- **可觀察性優於不透明性**：一切都必須透過 CLI 介面可檢查
- **簡潔性優於巧妙性**：從簡單開始，只有在證明必要時才增加複雜性
- **整合優於隔離**：在真實環境中測試，而不是人工環境中
- **模組化優於單體**：每個功能都是一個具有清晰邊界的函式庫

透過將這些原則嵌入到規格和規劃過程中，SDD 確保產生的程式碼不僅功能齊全——它還可維護、可測試且架構健全。核心準則將 AI 從程式碼產生器轉變為尊重和強化系統設計原則的架構合作夥伴。

## 轉型

這不是要取代開發人員或自動化創造力。這是透過自動化機械翻譯來放大人類能力。這是關於建立一個緊密的回饋循環，其中規格、研究和程式碼共同演進，每次迭代都帶來更深入的理解以及意圖與實作之間更好的對齊。

軟體開發需要更好的工具來維持意圖與實作之間的一致性。SDD 提供了透過產生程式碼而不是僅僅指導程式碼的可執行規格來實現這種一致性的方法。
