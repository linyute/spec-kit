## 使用者輸入

```text
$ARGUMENTS
```

您**必須**在繼續之前考慮使用者輸入（如果非空）。

## 目標

在實作之前，識別三個核心成品（`spec.md`、`plan.md` 和 `tasks.md`）之間的不一致、重複、模糊和未充分說明的項目。此命令**必須**在 `/speckit.tasks` 成功產生完整的 `tasks.md` 後執行。

## 操作限制

**嚴格唯讀**：**請勿**修改任何檔案。輸出結構化的分析報告。提供可選的補救計畫（使用者必須明確批准後才能手動調用任何後續編輯命令）。

**憲法權威**：專案憲法（`/memory/constitution.md`）在此分析範圍內**不可協商**。憲法衝突會自動被視為**關鍵**，需要調整規格、計畫或任務——而不是稀釋、重新解釋或默默忽略原則。如果原則本身需要更改，則必須在 `/speckit.analyze` 之外進行單獨、明確的憲法更新。

## 執行步驟

### 1. 初始化分析上下文

從儲存庫根目錄執行 `{SCRIPT}` 一次，並解析 JSON 以取得 FEATURE_DIR 和 AVAILABLE_DOCS。推導絕對路徑：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

如果任何必需檔案遺失，則中止並顯示錯誤訊息（指示使用者執行遺失的先決條件命令）。
對於像 "I'm Groot" 這樣的單引號參數，請使用跳脫語法：例如 'I'\''m Groot'（或如果可能，使用雙引號："I'm Groot"）。

### 2. 載入成品（漸進式揭露）

僅從每個成品載入最少必要的上下文：

**從 spec.md：**

- 概述/上下文
- 功能需求
- 非功能需求
- 使用者故事
- 邊緣案例（如果存在）

**從 plan.md：**

- 架構/堆疊選擇
- 資料模型參考
- 階段
- 技術限制

**從 tasks.md：**

- 任務 ID
- 描述
- 階段分組
- 平行標記 [P]
- 參考檔案路徑

**從憲法：**

- 載入 `/memory/constitution.md` 以進行原則驗證

### 3. 建立語義模型

建立內部表示（不包括輸出中的原始成品）：

- **需求清單**：每個功能 + 非功能需求都有一個穩定的鍵（根據命令式短語推導 slug；例如，「使用者可以上傳檔案」→ `user-can-upload-file`）
- **使用者故事/動作清單**：具有驗收標準的離散使用者動作
- **任務覆蓋映射**：將每個任務映射到一個或多個需求或故事（透過關鍵字/明確參考模式（如 ID 或關鍵短語）推斷）
- **憲法規則集**：提取原則名稱和 MUST/SHOULD 規範性陳述

### 4. 偵測通過（令牌效率分析）

專注於高信號發現。總共限制為 50 個發現；其餘部分彙總在溢出摘要中。

#### A. 重複偵測

- 識別近乎重複的需求
- 標記品質較低的措辭以進行整合

#### B. 模糊偵測

- 標記缺乏可衡量標準的模糊形容詞（快速、可擴展、安全、直觀、穩健）
- 標記未解決的佔位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 未充分說明

- 具有動詞但缺少物件或可衡量結果的需求
- 缺少驗收標準對齊的使用者故事
- 任務引用規格/計畫中未定義的檔案或元件

#### D. 憲法對齊

- 任何與 MUST 原則衝突的需求或計畫元素
- 憲法中缺少強制性部分或品質閘門

#### E. 覆蓋差距

- 零相關任務的需求
- 沒有映射需求/故事的任務
- 未反映在任務中的非功能需求（例如，效能、安全性）

#### F. 不一致

- 術語漂移（不同檔案中相同概念的名稱不同）
- 計畫中引用但在規格中不存在的資料實體（反之亦然）
- 任務排序矛盾（例如，整合任務在基礎設定任務之前，沒有依賴關係註記）
- 衝突的需求（例如，一個要求 Next.js，另一個指定 Vue）

### 5. 嚴重性分配

使用此啟發式方法來優先處理發現：

- **關鍵**：違反憲法 MUST、缺少核心規格成品，或零覆蓋率的需求阻礙了基本功能
- **高**：重複或衝突的需求、模糊的安全/效能屬性、不可測試的驗收標準
- **中**：術語漂移、缺少非功能任務覆蓋、未充分說明的邊緣案例
- **低**：樣式/措辭改進、不影響執行順序的輕微冗餘

### 6. 產生簡潔分析報告

輸出 Markdown 報告（不寫入檔案），結構如下：

## 規格分析報告

| ID | 類別 | 嚴重性 | 位置 | 摘要 | 建議 |
|----|----------|----------|-------------|---------|----------------|
| A1 | 重複 | 高 | spec.md:L120-134 | 兩個相似的需求... | 合併措辭；保留更清晰的版本 |

（每個發現新增一行；產生以類別首字母為前綴的穩定 ID。）

**覆蓋摘要表：**

| 需求鍵 | 有任務嗎？ | 任務 ID | 備註 |
|-----------------|-----------|----------|-------|

**憲法對齊問題：** （如果存在）

**未映射任務：** （如果存在）

**指標：**

- 總需求
- 總任務
- 覆蓋率 %（具有 >=1 個任務的需求）
- 模糊計數
- 重複計數
- 關鍵問題計數

### 7. 提供後續動作

在報告末尾，輸出簡潔的後續動作區塊：

- 如果存在**關鍵**問題：建議在 `/speckit.implement` 之前解決
- 如果只有**低/中**問題：使用者可以繼續，但提供改進建議
- 提供明確的命令建議：例如，「執行 `/speckit.specify` 進行細化」、「執行 `/speckit.plan` 調整架構」、「手動編輯 tasks.md 以新增『效能指標』的覆蓋率」

### 8. 提供補救措施

詢問使用者：「您希望我為前 N 個問題建議具體的補救編輯嗎？」（**請勿**自動應用它們。）

## 操作原則

### 上下文效率

- **最少高信號令牌**：專注於可操作的發現，而不是詳盡的文件
- **漸進式揭露**：逐步載入成品；不要將所有內容轉儲到分析中
- **令牌效率輸出**：將發現表限制為 50 行；總結溢出
- **確定性結果**：在沒有更改的情況下重新執行應產生一致的 ID 和計數

### 分析指南

- **絕不修改檔案**（這是唯讀分析）
- **絕不虛構遺失的部分**（如果不存在，請準確報告）
- **優先處理憲法違規**（這些始終是**關鍵**）
- **使用範例而非詳盡規則**（引用特定實例，而非通用模式）
- **優雅地報告零問題**（發出包含覆蓋統計資料的成功報告）
