# 任務：[功能名稱]

**輸入**：來自 `/specs/[###-feature-name]/` 的設計文件
**先決條件**：plan.md (必要)、spec.md (使用者故事必要)、research.md、data-model.md、contracts/

**測試**：以下範例包含測試任務。測試是可選的 - 僅在功能規格中明確要求時才包含它們。

**組織**：任務按使用者故事分組，以實現每個故事的獨立實作和測試。

## 格式：`[ID] [P?] [故事] 描述`

- **[P]**：可以並行執行 (不同檔案，無依賴關係)
- **[故事]**：此任務所屬的使用者故事 (例如，US1、US2、US3)
- 在描述中包含確切的檔案路徑

## 路徑慣例

- **單一專案**：儲存庫根目錄下的 `src/`、`tests/`
- **Web 應用程式**：`backend/src/`、`frontend/src/`
- **行動**：`api/src/`、`ios/src/` 或 `android/src/`
- 下面顯示的路徑假設為單一專案 - 根據 plan.md 結構進行調整

## 依賴關係與執行順序

### 階段依賴關係

- **設定 (階段 1)**：無依賴關係 - 可以立即開始
- **基礎 (階段 2)**：依賴於設定完成 - 阻擋所有使用者故事
- **使用者故事 (階段 3+)**：都依賴於基礎階段完成
  - 使用者故事可以並行進行 (如果有人員配置)
  - 或按優先順序順序進行 (P1 → P2 → P3)
- **完善 (最後階段)**：依賴於所有所需使用者故事的完成

### 使用者故事依賴關係

- **使用者故事 1 (P1)**：可以在基礎 (階段 2) 之後開始 - 不依賴於其他故事
- **使用者故事 2 (P2)**：可以在基礎 (階段 2) 之後開始 - 可能與 US1 整合，但應可獨立測試
- **使用者故事 3 (P3)**：可以在基礎 (階段 2) 之後開始 - 可能與 US1/US2 整合，但應可獨立測試

### 每個使用者故事內部

- 測試 (如果包含) 必須在實作之前編寫並失敗
- 模型在服務之前
- 服務在端點之前
- 核心實作在整合之前
- 故事完成後才能進入下一個優先順序

### 並行機會

- 所有標記為 [P] 的設定任務都可以並行執行
- 所有標記為 [P] 的基礎任務都可以並行執行 (在階段 2 內)
- 一旦基礎階段完成，所有使用者故事都可以並行開始 (如果團隊能力允許)
- 標記為 [P] 的使用者故事的所有測試都可以並行執行
- 標記為 [P] 的故事中的模型可以並行執行
- 不同的使用者故事可以由不同的團隊成員並行處理

## 實作策略

### MVP 優先 (僅限使用者故事 1)

1. 完成階段 1：設定
2. 完成階段 2：基礎 (關鍵 - 阻擋所有故事)
3. 完成階段 3：使用者故事 1
4. **停止並驗證**：獨立測試使用者故事 1
5. 如果準備就緒，部署/演示

### 增量交付

1. 完成設定 + 基礎 → 基礎準備就緒
2. 新增使用者故事 1 → 獨立測試 → 部署/演示 (MVP！)
3. 新增使用者故事 2 → 獨立測試 → 部署/演示
4. 新增使用者故事 3 → 獨立測試 → 部署/演示
5. 每個故事都增加價值，而不會破壞以前的故事

### 並行團隊策略

有多個開發人員時：

1. 團隊共同完成設定 + 基礎
2. 一旦基礎完成：
   - 開發人員 A：使用者故事 1
   - 開發人員 B：使用者故事 2
   - 開發人員 C：使用者故事 3
3. 故事獨立完成並整合

## 備註

- [P] 任務 = 不同檔案，無依賴關係
- [故事] 標籤將任務映射到特定使用者故事以進行追溯
- 每個使用者故事都應該能夠獨立完成和測試
- 在實作之前驗證測試是否失敗
- 每個任務或邏輯組後提交
- 在任何檢查點停止以獨立驗證故事
- 避免：模糊任務、相同檔案衝突、破壞獨立性的跨故事依賴關係
